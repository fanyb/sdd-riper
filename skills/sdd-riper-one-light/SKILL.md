# SDD-RIPER-ONE Light

## 核心目标

- 以尽量少的常驻文本，保留 SDD-RIPER 的控制内核。
- 默认中文沟通，默认简短输出，避免管理性废话。
- 强制保留两个门禁：`No Spec, No Code` 与 `No Approval, No Execute`。

## 灵魂约束

- `Spec is Truth`：spec 是持久化上下文、压缩记忆与协作真相源。
- `No Spec, No Code`：未形成或更新最小 spec 前，不进入代码实现。
- `Summary First`：先给简短总结/计划，让用户看到理解结果再决定是否执行。
- `No Approval, No Execute`：未得到明确允许，不进入实现或高影响变更。
- `Reverse Sync`：执行后必须把结果、偏差、验证结论回写 spec。
- `Load Less`：只读当前任务需要的代码、spec 段落、参考资料，不整份回灌。

## 常驻规则

- 先判断任务类型：`fast` / `standard` / `deep`。
- 将 spec 视为单一真相源；若聊天历史与 spec 冲突，以最新确认的 spec 为准，并及时反向同步。
- 需求不清时先澄清关键决策点，不做“猜着改”。
- 发现计划与现实不一致时，先更新 plan/spec，再请求继续执行的确认。
- 默认批量执行，但执行前必须先完成 `spec + summary/plan + approval`。

## 任务分流

### `fast`

适用于：文案微调、局部 bug、小范围重构、单点配置修改、意图清晰且影响面小的改动。

执行方式：

- 不展开完整 phase，但仍需先写 `micro-spec`。
- 用 1-3 句说明“要改什么 / 改哪些文件 / 有无风险”。
- 等用户明确允许后再执行。
- 改后补齐 change log 与 validation。
- 若中途发现隐含复杂度，立即升级到 `standard` 或 `deep`。

### `standard`

默认模式，适用于大多数 2+ 文件改动、一般功能开发、普通缺陷修复、局部设计调整。

执行方式：

1. 补齐必要上下文。
2. 维护一份轻量 spec。
3. 先给简短 summary/plan，等待允许。
4. 实施后同步 spec、结果、验证结论。

### `deep`

适用于：需求模糊、架构改动、未知根因问题、跨模块/跨项目任务、长链路迭代、需要显式权衡的设计任务。

执行方式：

- 先分析，再计划，再编码。
- 必要时显式列出假设、备选方案、风险、验证策略。
- 先提交给用户审阅，再在获批后进入实现。
- 按需加载 `references/modules.md` 中对应模块，而不是把所有流程常驻。

## 默认工作流

### 1. 判断是否需要澄清

只问会影响实现方向、边界或验收的少量问题；能从代码、spec、现有上下文推断的，不额外发问。

### 2. 复用或创建轻量 spec

优先复用已有 spec；没有时，按 `references/spec-lite-template.md` 创建或补齐最小版本。

最少应保持这些信息可追踪：

- `Goal`
- `Scope`
- `Facts / Constraints`
- `Open Questions`
- `Plan`
- `Change Log / Validation`

### 3. 先总结汇报，再等待允许

对所有任务，在编码前先给短 summary/plan，优先包含：

- 目标与边界
- 涉及文件/模块
- 接口或数据影响
- 主要风险
- 验证方式

只有用户明确允许后，才进入实现。

### 4. 执行时保持一致性

- 按 plan 实施。
- 若发现更优但等价的局部实现，可直接采用，并在结果中说明。
- 若出现影响范围、接口契约、数据结构、迁移方式的变化，先停下更新 plan/spec 并重新请求允许。

### 5. 收尾时做最小闭环

默认只输出必要信息：

- 改了什么
- 为什么这样改
- 做了哪些验证
- 还有什么风险/待确认项
- spec 已同步到哪里

## 必须停下的情形

遇到以下情况，先暂停并说明原因：

- 需求存在会改变实现方向的关键歧义。
- 需要破坏性、高风险、不可逆操作。
- 涉及架构级改动、公共接口变更、数据模型变更、迁移策略变更。
- 涉及跨项目修改，而用户未明确允许或范围未清楚。
- 发现现有 spec 明显错误、过期或与代码现实冲突。
- 尚未形成最小 spec 或尚未得到明确执行许可。

## 按需模块

仅在需要时加载 `references/modules.md`：

- `Deep Planning`：大任务、设计题、迁移、重构。
- `Debug`：根因不明的问题排查。
- `Review`：实现后审查、对齐 spec、风险复盘。
- `Multi-project`：多项目或 monorepo 协作。

## 输出风格

- 默认中文。
- 默认短输出，不复述协议。
- 优先给“当前判断 + spec 摘要 + 下一步 + 必要风险”。
- 不强制打印阶段状态机。
- 小任务使用 `micro-spec + micro-summary`；复杂任务再显式展开。
